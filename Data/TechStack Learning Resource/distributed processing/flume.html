  2. Data Flow Model:

    • A Flume event consists of a byte payload and an optional set of string attributes.
    • A Flume agent (a JVM process) hosts components through which events flow from an external source to the next destination (hop).
    • A Flume source consumes events delivered by an external source (e.g., a web server). The source formats the events for recognition by the target Flume source (e.g., Avro Flume source or Thrift Flume source).
    • Events received by a Flume source are stored in one or more channels (passive stores).
    • The sink removes events from the channel and either:
      • Stores them in an external repository (e.g., HDFS via Flume HDFS sink).
      • Forwards them to the next Flume agent (next hop) in the flow.
    • Sources and sinks within an agent run asynchronously, with events staged in the channel.
    • Flume allows multi-hop flows, fan-in, fan-out, contextual routing, and backup routes for failed hops.

  3. Reliability:

    • Events are staged in a channel on each agent.
    • Events are delivered to the next agent or terminal repository (e.g., HDFS) in the flow.
    • Events are removed from a channel only after they are stored in the channel of the next agent or terminal repository.
    • This single-hop message delivery semantics ensure end-to-end reliability.
